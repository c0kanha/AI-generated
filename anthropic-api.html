Anthropic API

The cloud service that sits between the harness on your machine and the Claude LLM on Anthropic's GPUs.

What Is It?

The Anthropic API is a web service (a cloud endpoint) that accepts HTTP requests and returns AI-generated responses. It is the front door to Anthropic's AI models.

It is NOT the AI itself. It is a server application that:

  • Receives your request over the internet
  • Validates your API key and checks your billing
  • Passes your prompt to the Claude LLM running on GPUs
  • Streams the LLM's output back to you

Analogy: A restaurant host

You (the harness) walk into a restaurant (Anthropic). The host (API) checks your reservation (API key), takes your order (prompt), passes it to the chef (LLM), and brings back the dish (response). The host doesn't cook — they just manage the interaction. The chef never meets you directly.

Where Does It Run?

Detail Answer
Location Anthropic's cloud servers (data centers, likely US-based)
Endpoint https://api.anthropic.com/v1/messages
Protocol HTTPS (encrypted) — REST API with JSON payloads
Authentication API key sent in the x-api-key header
Access Public internet — anyone with a valid API key can call it

What Goes In (The Request)

The harness sends a JSON request that looks like this:

// Simplified — actual request has more fields
{
  "model": "claude-opus-4-6",
  "max_tokens": 16000,
  "system": "You are Claude Code, Anthropic's CLI... [~4000 words of instructions]",
  "tools": [
    {
      "name": "Read",
      "description": "Reads a file from the local filesystem...",
      "input_schema": { "type": "object", "properties": { "file_path": {...} } }
    },
    {
      "name": "Bash",
      "description": "Executes a bash command...",
      "input_schema": { ... }
    }
    // ... 13 more tools
  ],
  "messages": [
    { "role": "user", "content": "can convert scan.pdf to html" },
    { "role": "assistant", "content": [{ "type": "tool_use", "name": "Glob", ... }] },
    { "role": "user", "content": [{ "type": "tool_result", ... }] }
    // ... full conversation history
  ]
}
Field What it contains Who created it
model Which Claude model to use (e.g., claude-opus-4-6) Harness (from your settings)
system System prompt: rules, behavior instructions, environment info Harness (built-in + CLAUDE.md)
tools JSON schemas defining every tool the LLM can call Harness (built-in)
messages Full conversation: your messages, LLM responses, tool calls, tool results, images Harness (accumulated over the session)
max_tokens Maximum length of the LLM's response Harness

The full conversation is sent every time

The API is stateless — it doesn't remember previous calls. Every request must include the entire conversation from the beginning. If you've had 20 back-and-forths, all 20 are sent again. This is why the harness manages conversation history — the API just processes whatever it receives, with no memory of what came before.

What Comes Out (The Response)

The API returns a JSON response. There are two possible shapes:

Shape 1: Text response (task is done)

{
  "role": "assistant",
  "stop_reason": "end_turn",
  "content": [
    {
      "type": "text",
      "text": "Done. Created scan.html with all 14 pages..."
    }
  ]
}

When the harness sees stop_reason: "end_turn" and text content, it displays the text to you. The conversation turn is complete.

Shape 2: Tool call response (LLM wants to do something)

{
  "role": "assistant",
  "stop_reason": "tool_use",
  "content": [
    {
      "type": "text",
      "text": "I need to install poppler to read the PDF."
    },
    {
      "type": "tool_use",
      "id": "toolu_01ABC123",
      "name": "Bash",
      "input": {
        "command": "brew install poppler",
        "description": "Install poppler for PDF rendering"
      }
    }
  ]
}

When the harness sees stop_reason: "tool_use", it knows the LLM wants to execute a tool. The harness extracts the tool name and parameters, executes the tool locally, and sends the result back as a new API request.

What the API Does Internally

When a request arrives, the API performs these steps:

Step What happens Purpose
1 Validate the API key in the x-api-key header Authentication: is this a real, active account?
2 Check rate limits and usage quotas Prevent abuse and manage capacity
3 Validate the request format (correct JSON structure, valid model name) Reject malformed requests early
4 Tokenize the input (convert text to token IDs the model understands) The LLM processes tokens, not raw text
5 Check the prompt cache — has this exact prefix been seen recently? Prompt caching: reuse prior computation to save time and cost
6 Route the request to available GPU workers running the Claude model Load balancing across GPU clusters
7 The LLM generates output tokens one by one This is the actual "thinking" step
8 Stream tokens back to the caller as they're generated You see text appear progressively, not all at once
9 Log usage (input tokens, output tokens) for billing You pay per token processed

Prompt Caching

Because the full conversation is sent every time, the API uses prompt caching to avoid redundant computation:

How caching works

On your first API call, the system prompt + tool definitions are processed from scratch. The result is cached for 5 minutes. On the next call (which includes the same system prompt), the API says "I've already processed this prefix" and skips straight to the new content. This makes subsequent calls much faster and cheaper.

If more than 5 minutes pass between calls, the cache expires and the full prompt is reprocessed. This is why long pauses between actions cost more.

What the API Does NOT Do

It does NOT... Why not
Think or reason The API is just a web server. The LLM (running on GPUs behind the API) does the thinking.
Access your files or machine The API only receives what the harness sends it. It has no connection to your computer.
Execute tool calls The LLM generates tool call instructions. The API passes them to the harness. The harness executes them on your Mac.
Remember previous requests Every request is independent. Conversation continuity is the harness's job.
Store your data long-term Your prompts and responses are processed and discarded. They are not used for training (per Anthropic's API policy).
Run on your machine It runs on Anthropic's cloud servers. Your data travels over the internet to reach it.

Billing: How You Pay

The API charges based on tokens (pieces of words):

What Description You pay for
Input tokens Everything sent TO the API: system prompt, conversation history, tool definitions, images Every token, every call (reduced if cached)
Output tokens Everything the LLM generates: text responses, tool call instructions, HTML code Every token generated
Cached input Portions of the input that match the prompt cache Reduced rate (much cheaper than uncached)

Why long conversations get expensive

Remember: the full conversation is sent on every API call. A 50-message conversation means each new call includes all 50 previous messages as input tokens. The input cost grows with every turn. Prompt caching helps, but only for the unchanging prefix.

Key Takeaways

  • The API is a web service — a server that accepts HTTP requests and returns JSON responses
  • It runs on Anthropic's cloud, not your machine
  • It is stateless — no memory between calls. The harness resends everything each time.
  • It does not think — it routes requests to the LLM which does the thinking
  • It does not execute tools — it passes tool call instructions back to the harness
  • It handles auth, billing, caching, and load balancing
  • You pay per token (input and output), and costs grow as conversations get longer
