Claude Code Harness

The local program on your machine that connects you to the AI and executes actions on your behalf.

What Is It?

The Claude Code Harness is a program that runs on your computer. When you type claude in your terminal, you're launching the harness. It is not the AI itself — it is the messenger and executor that sits between you and the AI.

Analogy: A personal assistant

Imagine you hire an assistant. You tell them what you need. They walk to the expert (the LLM), relay your request, bring back the expert's advice, and then physically do the work (open files, run commands, etc.). The expert never leaves their office — your assistant does all the legwork. The harness is that assistant.

Where Does It Run?

Detail Answer
Location Your local machine (Mac, Windows, or Linux)
How to start Type claude in terminal, or open Claude Code desktop app, or use VS Code / JetBrains extension
Written in TypeScript / Node.js
Runs as A local process on your machine, with access to your file system and shell
Needs internet? Yes — to communicate with the Anthropic API. But all tool execution is local.

What Does It Do?

The harness has 7 main jobs:

1. Receives your input

When you type a prompt, the harness captures your text from the terminal. It also detects slash commands (like /help), file references, and image paths.

2. Packages the API request

Your prompt alone isn't enough. The harness builds a complete API payload containing:

What Where it comes from Why the LLM needs it
System prompt Built into the harness + CLAUDE.md files Tells the LLM how to behave, what rules to follow, what tools are available
Tool definitions Built into the harness (JSON schemas) The LLM needs to know what tools exist, what parameters they take, and what they do
Conversation history Stored in harness memory (this session) The LLM has no memory between calls — the harness replays the full conversation each time
Your new message What you just typed The actual request to respond to
Previous tool results Captured by the harness from local execution The LLM needs to see what happened when tools were run

3. Sends the request to Anthropic API

The harness sends the full payload as an HTTPS POST request to api.anthropic.com. This is an internet call — your data leaves your machine and travels to Anthropic's servers.

4. Receives the LLM's response

The API sends back a JSON response. The harness parses it and determines: is this plain text (display to user) or tool calls (execute locally)?

5. Executes tool calls locally

This is the harness's most important job. When the LLM says "run this bash command" or "read this file," the harness actually does it on your machine.

Tool What the harness does locally
Read Opens a file on your disk, reads its contents (or renders PDF/image), returns the content to the LLM
Write Creates or overwrites a file on your disk with content the LLM generated
Edit Finds a specific string in a file on disk and replaces it with new text
Bash Spawns a child process on your machine, runs a shell command, captures stdout/stderr
Glob Searches your file system for files matching a pattern (like **/*.pdf)
Grep Searches file contents on your disk using regular expressions
Agent Spawns a sub-conversation with the LLM (a separate API call chain) for parallel work

The tools run on YOUR machine, not on Anthropic's servers

When the LLM says Bash("brew install poppler"), it's your Mac that runs brew install poppler. Anthropic's servers never touch your file system. The harness is the bridge that makes this possible.

6. Enforces permissions

Before executing a tool call, the harness checks your permission settings:

  • Auto-allowed: Some tools (like Read, Glob, Grep) may be pre-approved — the harness runs them immediately
  • Requires approval: Dangerous tools (like Bash, Write) show a prompt: "Claude wants to run [command]. Allow?" — you press Y or N
  • Denied: If you press N, the harness tells the LLM "the user denied this tool call" and the LLM must find another approach

You are always in control

The harness never runs dangerous commands without your approval. Even if the LLM generates a rm -rf / command, the harness would show it to you first and wait for your explicit approval. You are the gatekeeper.

7. Displays output to you

When the LLM sends back plain text (not tool calls), the harness renders it in your terminal with Markdown formatting — bold, code blocks, headers, lists, etc.

What It Does NOT Do

It does NOT... Why not
Think, reason, or make decisions It has no intelligence. It just follows instructions from the LLM.
Generate text, code, or creative content All generated content comes from the LLM. The harness just passes it through.
Understand your intent It doesn't interpret your prompt. It forwards it to the LLM for interpretation.
Store long-term memory Conversation memory only lasts for the current session. The harness doesn't remember previous sessions (except through memory files on disk).
Run on Anthropic's servers It runs entirely on your machine. It's a local Node.js process.

The harness sits in the middle and talks to three parties:

  1. You — receives your input, shows you output, asks for permission
  2. Anthropic API — sends prompts, receives LLM responses (over HTTPS)
  3. Your Mac — executes tools, reads/writes files, runs commands (local)

What Happens Inside the Harness (Step by Step)

Here's the internal process on every turn:

Step What happens inside the harness
1 Capture your input from stdin (terminal keyboard)
2 Append your message to the conversation history array
3 Build the API payload: system prompt + tool schemas + full conversation history
4 Send HTTPS POST to api.anthropic.com/v1/messages with your API key
5 Receive streaming response (tokens arrive one by one)
6 Parse response: is it text content or tool-use content?
7a If text: render Markdown and display in terminal. Done for this turn.
7b If tool call: check permission settings
8 If permission needed: display prompt to user, wait for Y/N
9 If approved: execute the tool locally (read file, run bash, etc.)
10 Capture the tool's output (file content, stdout, error messages, images)
11 Append both the LLM's tool call and the tool result to conversation history
12 Go back to step 3 (send updated conversation to API for the next LLM turn)

The loop continues until the LLM stops calling tools

Steps 3–12 repeat for every tool call. When the LLM finally responds with plain text (no tool calls), the loop ends and the text is displayed to you. A single prompt from you might trigger 1 loop iteration or 20 — depending on how complex the task is.

Session Memory vs Persistent Memory

Type How it works Lifespan
Session memory The harness keeps the full conversation history in RAM and replays it on every API call Dies when you close the terminal / exit Claude Code
Persistent memory The LLM can ask the harness to write memory files to ~/.claude/projects/.../memory/ Survives across sessions (files on disk)
CLAUDE.md Project-level instructions the harness includes in the system prompt automatically Permanent (part of the project)

The LLM has no memory of its own

Every API call, the LLM starts completely fresh. It doesn't remember the last call. The harness creates the illusion of memory by resending the full conversation every time. This is why long conversations get slower — the payload keeps growing.

Key Takeaways

  • The harness is a local program on your machine, not a cloud service
  • It is the only actor that can touch your files, run commands, and access your system
  • It has no intelligence — it just relays messages and executes instructions
  • It enforces permissions — you approve or deny every risky action
  • It manages conversation state — the LLM is stateless, the harness provides continuity
  • Without the harness, the LLM would be a brain in a jar with no way to interact with the real world
