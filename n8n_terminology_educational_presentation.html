n8n Terminology — Explained Clearly

This version focuses on understanding, not just naming terms. Special attention is given to the three trigger types that usually confuse beginners: Webhook, Schedule, and App Event.
Big idea
In n8n, a workflow runs when a trigger fires. The trigger is the start button. Everything after that is a series of nodes that read data, change data, or send data somewhere else.
Workflow
The full automation.
Node
One step inside the automation.
Trigger
The event or condition that starts the workflow.

Foundation

First understand these 4 words

Workflow

The complete automation you build. It connects multiple nodes to move from input to output.

Node

A single step inside the workflow. One node may fetch data, one may transform it, and another may send it out.

Trigger node

The special starting node. Without a trigger, the workflow does not know when to run automatically.

Execution

One actual run of the workflow. If the workflow runs 50 times, that means 50 executions.

Core definition structure used in the official n8n glossary and executions docs.

Most important concept

What does “Webhook” mean?

Webhook = “someone sends data to a URL, and that starts n8n”
A webhook is an HTTP endpoint created by n8n. Another app or service calls that URL when something happens, and that call starts your workflow.
External system
A form, payment app, CRM, website, or custom app sends an HTTP request.
→
Webhook URL
n8n exposes a URL. When that URL receives the request, the trigger fires.
→
Workflow execution
n8n reads the incoming data and runs the next nodes.
Use webhook when: another system can send an HTTP request, but you do not want polling or there is no dedicated trigger node.
Example: Stripe payment completed → call webhook → send invoice email → update database.

Second trigger type

What does “Schedule” mean?

Schedule = “run this workflow at fixed time intervals”
The Schedule Trigger runs the workflow based on time — every minute, every hour, every Monday, or a custom cron pattern.

How it starts

n8n itself starts the workflow because the time rule matches.

Typical use

Daily reports, nightly syncs, hourly checks, weekly reminders, batch jobs.

Important detail

Time zone matters. If time zone is wrong, the workflow runs at the wrong hour.

Example: Every weekday at 8:00 AM → pull sales metrics → send summary to Slack.
Common misunderstanding: schedule is not waiting for an app event. It runs because the clock says it is time.

Third trigger type

What does “App event” mean?

App event = “a specific app tells n8n that something happened inside that app”
This is usually an app-specific trigger node such as GitHub Trigger, Slack Trigger, Gmail Trigger, Salesforce Trigger, and so on. The app has named events, and the trigger runs when one of those events happens.

Examples of app events

  • GitHub: push, pull request, release
  • Slack: new message or mention
  • Salesforce: record created or updated
  • Calendars: event created or changed

How it differs from webhook

Webhook is generic: any system can call a URL. App event triggers are prebuilt for a specific app and expose named events from that app.

Working with data

Expressions, $json, Variables, and Code node

Expression Small inline logic written inside {{ ... }}. Use it when a field should be filled dynamically from previous data.
$json The current item’s JSON data. Example: read a name, email, city, or status from the current input item.
Variables Reusable read-only values stored in n8n. Good for constants like environment names, IDs, or shared configuration values.
Code node A full step where you write JavaScript or Python when simple expressions are not enough.
When to use expression Pull one value, do a light calculation, or keep logic close to one field.
When to use code node Complex transformation, loops, restructuring arrays, custom algorithms, or processing many items together.
Slide 7

Access, security, and organization

Credentials, Projects, Activation, and Executions

Credentials

Saved authentication details used to connect nodes to external apps and APIs securely.

Projects

A way to group workflows and credentials and manage user access by project.

Activation / Production mode

When a workflow is active, its triggers can run automatically. Inactive workflows are mainly for development and testing.

Execution history

Each run is stored as an execution. You inspect executions to debug failures, review outputs, and audit what happened.
