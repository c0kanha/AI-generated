Claude LLM

The language model that does the thinking — what it is, what it can do, what it cannot do, and how it actually works inside.

What Is It?

LLM stands for Large Language Model. Claude is Anthropic's LLM. It is a massive mathematical function — a neural network with billions of parameters — that takes text (and images) as input and generates text as output.

That's it. That's all it does. Text in, text out.

Analogy: A brilliant expert locked in a room

Imagine the world's most knowledgeable expert locked in a sealed room with no windows, no phone, and no internet. You can only communicate by sliding notes under the door. You write a question, slide it in. They write an answer, slide it back. They can't see your screen, can't touch your files, can't browse the web. They can only read what you give them and write back.

But their answers are extraordinarily good — because they've read billions of documents during their training and can reason about almost anything.

Where Does It Run?

Detail Answer
Location Anthropic's data centers, on clusters of GPUs (Graphics Processing Units)
Hardware Thousands of high-end GPUs (likely NVIDIA H100 or similar)
Size Billions of parameters (exact number not public). The model weights alone are hundreds of gigabytes.
Runs on your machine? No. It is far too large to run on a laptop or desktop. It requires specialized hardware.
Who triggers it The Anthropic API loads your prompt onto the GPUs and runs the model

What It Can and Cannot Do

CAN do (its own abilities)

  • Read and understand text in any language
  • Read and understand images (photos, screenshots, diagrams, charts)
  • Generate text: prose, code, JSON, HTML, SQL, any text format
  • Reason: plan multi-step solutions, weigh options, diagnose errors
  • Follow instructions: obey system prompts, respect constraints
  • Decide which tool to call and with what arguments
  • Hold context: work with ~200K tokens of conversation at once

CANNOT do (needs the harness)

  • Read files from your computer
  • Write or create files
  • Run commands or install software
  • Access the internet or browse websites
  • See your screen
  • Remember previous conversations
  • Execute any action in the real world
  • Learn from this conversation (its weights are frozen)

The "cannot" list is fundamental, not a limitation to be fixed

The LLM literally has no mechanism to access your file system. It's not a security restriction — it's architecture. The model is a function: f(tokens) → tokens. There is no file-system driver, no network socket, no process spawner in its architecture. The harness provides all of these externally.

How It Works Inside

When the LLM receives your prompt, here's what happens at a technical level:

Step 1: Tokenization

Your text is broken into tokens — small pieces of words. For example:

Your text Tokens (approximate)
can convert scan.pdf to html ["can", " convert", " scan", ".", "pdf", " to", " html"] — 7 tokens
brew install poppler ["bre", "w", " install", " pop", "pler"] — 5 tokens

Each token is converted to a number (token ID). The model only works with numbers, not letters.

Step 2: Processing through the neural network

The token IDs pass through the neural network — layer by layer, through billions of mathematical operations. Each layer transforms the data, gradually building up an understanding of:

  • What the words mean individually
  • How they relate to each other
  • What the user is asking for
  • What an appropriate response would be
  • What tools would be helpful to call

The "attention" mechanism

The core innovation in LLMs is called attention. It allows every token in the input to look at every other token and determine how relevant they are to each other. When the model reads "convert scan.pdf to html," the word "convert" attends to "pdf" and "html" to understand the transformation being requested. This happens across all layers simultaneously, building increasingly abstract representations of meaning.

Step 3: Generate output one token at a time

The model doesn't generate its entire response at once. It produces one token at a time:

  1. Process all input tokens through the network
  2. The network outputs a probability distribution over all possible next tokens (~100,000 options)
  3. Select the most likely token (with some randomness controlled by "temperature")
  4. Append that token to the output
  5. Feed the entire sequence (input + output so far) back through the network
  6. Generate the next token
  7. Repeat until the model produces a stop token or hits the max length

Token-by-token generation

Input: "can convert scan.pdf to html"
→

→
Output token 1: "Let"
Input + "Let"

This is why AI responses stream in gradually

When you see text appearing word by word in your terminal, that's not an animation — the model is literally generating one token at a time and streaming each one as it's produced. The first word is generated before the model knows what the last word will be.

How It Decides to Call a Tool

The LLM wasn't born knowing about tools. During training, it was taught a specific format for tool calls. When it decides a tool would be helpful, it generates structured JSON instead of plain text:

What the LLM sees in its input What the LLM generates
Tool definitions (JSON schemas) listing available tools, their parameters, and descriptions A tool_use content block with the tool name and arguments as JSON

The LLM doesn't "call" the tool. It generates text that describes a tool call. The harness parses this text and does the actual calling.

// The LLM generates this as its output tokens:
{
  "type": "tool_use",
  "name": "Bash",
  "input": {
    "command": "brew install poppler"
  }
}

// The LLM has NO IDEA if this command will actually run.
// It doesn't have a terminal. It just produces text that
// DESCRIBES what it wants to happen.

Tool use is just structured text generation

When the LLM "calls" Bash("brew install poppler"), it is not executing anything. It is generating the text {"name": "Bash", "input": {"command": "brew install poppler"}}. The harness reads this text, recognizes the structured format, and executes the command on your Mac. If the harness didn't exist, the tool call text would just sit there, doing nothing.

How It "Sees" Images (Vision)

Claude is a multimodal model — it can process images alongside text.

Step What happens
1 The harness reads a PNG file from your disk and encodes it as base64 data
2 The base64 image is included in the API request as an image content block
3 The API passes the image to the LLM
4 The LLM's vision encoder converts the image pixels into the same kind of token representations the text encoder produces
5 These visual tokens are processed alongside text tokens through the same neural network
6 The model can now "see" the image and reference its contents when generating a response

This is how the LLM read the 14 scanned PDF pages — it received 14 images and visually interpreted the text, tables, diagrams, and layout in each one.

What Are "Parameters" and "Weights"?

When people say the model has "billions of parameters," they mean:

Parameters = the model's learned knowledge

Parameters (also called weights) are numbers stored in the neural network. Each parameter is a single decimal number (like 0.0342 or -1.2847). There are billions of them.

During training, these numbers were adjusted to encode patterns from the training data. The parameters collectively encode the model's "knowledge" — not as facts in a database, but as statistical patterns of language. The model doesn't store "brew install poppler" as a fact; instead, it has learned that when someone encounters a "not installed" error on macOS, the pattern brew install [package] is statistically the most helpful next step.

Concept What it means Analogy
Parameters Billions of numbers that define the model's behavior Like the synaptic connections in a brain — each one is tiny, but together they create intelligence
Training The process of adjusting parameters using training data (done once, before deployment) Like going to school for years — learning happens during training, not during conversations
Inference Using the trained model to generate responses (what happens when you chat) Like taking a test — using what you learned, but not learning new things
Context window The maximum number of tokens the model can process at once (~200K for Claude) Like short-term memory — how much you can hold in your head at one time
Tokens Pieces of words (~4 characters each on average) Like syllables — the smallest units the model reads and writes

Training vs. Using (Inference)

Training (already done)

  • Happened before you ever used Claude
  • Took weeks/months on thousands of GPUs
  • Processed billions of documents (web pages, books, code, etc.)
  • Adjusted billions of parameters
  • Cost millions of dollars
  • The model learned during this phase

Inference (what happens now)

  • Happens every time you send a message
  • Takes seconds on a few GPUs
  • Processes only your conversation (not new training data)
  • Parameters are frozen — they don't change
  • Costs cents to dollars per conversation
  • The model uses what it learned, but does not learn new things

Your conversations do not train the model

When you chat with Claude through the API, your messages are processed and discarded. The model's parameters are not updated. It doesn't "learn" from your conversation. If you teach it something in one session, it won't remember it in the next. The harness creates the illusion of memory by replaying the conversation history, but the model itself is the same before and after your interaction.

The Model Family

Claude comes in different sizes, each trading off capability for speed and cost:

Model Strengths When to use
Claude Opus 4.6 Most capable. Best reasoning, coding, and analysis. Handles complex multi-step tasks. Complex tasks, code generation, deep analysis (this is what generated scan.html)
Claude Sonnet 4.6 Very capable. Good balance of quality and speed. Faster and cheaper than Opus. Most everyday tasks, when speed matters more than maximum quality
Claude Haiku 4.5 Fastest and cheapest. Good for simple tasks. Quick lookups, simple questions, high-volume tasks

All three share the same architecture but differ in the number of parameters. More parameters = more capability = slower = more expensive.

What Happened Inside Claude During the PDF Task

Here's what the LLM actually did (internally) for the scan.pdf conversion:

Round Input the LLM received What the LLM's neural network computed Output the LLM produced
1 System prompt + your message "can convert scan.pdf to html" Understood intent: convert a file. Decided to check if it exists first. Selected the Glob tool from its available tools. Text + Glob("**/scan.pdf")
2 Previous + Glob result: "scan.pdf found" File exists. It's a PDF. The Read tool can handle PDFs. Try reading pages 1-5. Read("scan.pdf", pages="1-5")
3 Previous + error: "pdftoppm not installed" Error mentions poppler-utils. System is macOS (from system prompt: "Platform: darwin"). Homebrew is the standard package manager on macOS. Correct command: brew install poppler. Text + Bash("brew install poppler")
6 Previous + pdftotext result: (empty) No text extracted = scanned image PDF. Strategy: convert pages to images with pdftoppm (which works in Bash), then use Read to see the images. Chose 200 DPI as balance of quality/speed. Bash("pdftoppm -png -r 200 ...")
9 Previous + 7 page images (raw pixels) Vision encoder processed each image. Recognized: titles, paragraphs, tables, code blocks, a JSON config, a file tree diagram, bullet lists. Held all content in context. Request to read 7 more pages
11 Previous + all 14 page images Full document understood. Planned HTML structure: 10 sections, matching CSS from prior conversation (purple theme). Generated 476 lines of HTML/CSS token by token. Write("scan.html", content=...)

Common Misconceptions

Misconception Reality
"The AI accessed my file system" The LLM never accessed your files. The harness read files and sent the contents to the LLM as text/images.
"The AI ran a bash command" The LLM generated text describing a command. The harness ran the command on your Mac.
"The AI learned from my conversation" The model's weights are frozen during inference. It doesn't learn. It processes each request from scratch.
"The AI remembers our last conversation" The LLM has zero memory between sessions. The harness resends the full conversation history, creating an illusion of memory.
"The AI is running on my computer" The LLM runs on Anthropic's GPUs in the cloud. Only the harness runs on your machine.
"The AI thinks like a human" The LLM predicts the most likely next token based on statistical patterns. It doesn't have consciousness, beliefs, or experiences. It produces remarkably human-like output, but the mechanism is fundamentally different.
"The AI copies from its training data" The LLM doesn't store or retrieve documents. It generates new text based on learned patterns. The 476-line HTML was genuinely novel — no identical file existed in its training data.

Key Takeaways

  • The LLM is a mathematical function: text/images in, text out. Nothing more.
  • It runs on Anthropic's GPUs, not your machine
  • It generates output one token at a time, predicting the most likely next word
  • It has no access to your files, your network, or the internet
  • It does not learn from your conversations — its parameters are frozen
  • Tool calls are just structured text that the harness parses and executes
  • Vision works by converting images into the same kind of internal representations as text
  • Its "knowledge" comes from training on billions of documents, encoded as billions of numerical parameters
  • It is extraordinarily capable at language tasks, but it is not sentient, not conscious, and not a database
